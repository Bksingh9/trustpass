# TRUSTPASS Schema

## Purpose

This document defines the PostgreSQL schema for the TRUSTPASS MVP. It is the source of truth for the first Alembic migrations and the SQLAlchemy 2.x model layer.

TRUSTPASS is multi-tenant by organization. Vendors, buyers, internal reviewers, and super admins all operate through organizations and memberships. Supabase Auth owns credential storage and session identity. TRUSTPASS stores application users, organization membership, role data, compliance workflows, trust scoring evidence, buyer interactions, billing records, and audit history.

## Global Conventions

- Database: PostgreSQL.
- Primary keys: UUID columns named `id`, generated by the application or database default.
- Timestamps: `created_at` and `updated_at` on mutable business tables, stored as timezone-aware UTC timestamps.
- Naming: snake_case table and column names.
- Tenant isolation: tenant-owned data includes `organization_id` unless the record is a global catalog item or a cross-organization interaction.
- Auth boundary: Supabase Auth stores passwords, sessions, MFA, and login credentials. TRUSTPASS never stores passwords.
- IDs from Supabase Auth: `users.auth_subject_id` stores the Supabase user identifier.
- Statuses: use PostgreSQL enums or SQLAlchemy enum-backed strings, with Alembic migrations owning enum lifecycle.
- Auditability: important state changes write to `audit_events`; user-facing activity can also write to `activity_logs`.
- Soft delete: use only where recovery or historical integrity matters. Prefer status fields for workflow records.
- Private files: Postgres stores metadata and object keys only. Raw files live in local storage for development or S3-compatible storage in production.

## Tenant And Access Rules

- Vendor-owned records are anchored to the vendor organization through `organization_id`.
- Buyer-owned records are anchored to the buyer organization through `organization_id`.
- Internal admin users belong to an internal organization and receive admin permissions through `memberships`.
- Super admin permissions are granted only through `memberships.role = 'super_admin'` on an internal organization.
- Cross-organization workflows use explicit relationship tables:
  - `shortlists` links buyer organizations to vendor organizations.
  - `buyer_vendor_requests` links buyer organizations to vendor organizations.
  - `verification_requests` links a vendor organization to reviewer users and internal review status.
- Buyer-facing reads must use buyer-safe projections from vendor profile, badges, approved status, categories, coverage, and public case-study/reference summaries. Buyers must not receive private document object keys, reviewer notes, or sensitive uploaded files.
- Admin-facing reads can include verification evidence, document metadata, score breakdowns, and audit history, subject to admin role checks.

## Core Identity And RBAC

### `users`

Purpose: Application profile mapped to a Supabase Auth subject.

Key columns:

- `id uuid primary key`
- `auth_subject_id text not null`
- `email citext not null`
- `full_name text`
- `phone text`
- `avatar_url text`
- `status user_status not null default 'active'`
- `last_seen_at timestamptz`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Relationships:

- One user can have many `memberships`.
- One user can create or review documents, requests, notes, logs, and audit events.

Constraints and indexes:

- Unique: `auth_subject_id`
- Unique: `email`
- Index: `status`
- Index: `last_seen_at`

Soft delete: No. Disable access with `status = 'disabled'` so historical records keep valid authors.

### `roles`

Purpose: Role catalog for application permissions and future custom role expansion.

Key columns:

- `id uuid primary key`
- `name membership_role not null`
- `display_name text not null`
- `description text`
- `permissions text[] not null default '{}'`
- `is_system boolean not null default true`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Relationships:

- Referenced by `memberships.role_id` where explicit role records are needed.

Constraints and indexes:

- Unique: `name`
- Index: `is_system`

Soft delete: No. System roles are stable catalog data.

### `memberships`

Purpose: Connect users to organizations and define their role in each tenant.

Key columns:

- `id uuid primary key`
- `organization_id uuid not null references organizations(id)`
- `user_id uuid not null references users(id)`
- `role membership_role not null`
- `role_id uuid references roles(id)`
- `status membership_status not null default 'active'`
- `invited_by_user_id uuid references users(id)`
- `invited_at timestamptz`
- `accepted_at timestamptz`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Relationships:

- Many memberships belong to one `organization`.
- Many memberships belong to one `user`.

Constraints and indexes:

- Unique: `(organization_id, user_id)`
- Index: `(user_id, status)`
- Index: `(organization_id, role, status)`
- Check: `role` must be compatible with `organizations.type`.

Soft delete: No. Use `status = 'removed'` to preserve history.

- `deleted_at timestamptz`

- Belongs to one vendor organization.
- Belongs to one document type.
- Can replace another document through `replaces_document_id`.
- Drives `verification_checks` and `vendor_checklist_progress`.

- Index: `(organization_id, document_type_id, status)`
- Index: `(status, expires_at)`

Access notes:

- Raw file URLs are generated by the storage service as signed URLs.
- Buyer APIs must not expose `storage_bucket`, `storage_object_key`, checksum, rejection notes, or private review metadata.
- Admin APIs may expose metadata and signed view/download URLs according to RBAC.

### `trust_score_snapshots`

Key columns:

- `id uuid primary key`

- `score integer not null`
- `trust_level text not null`
- `breakdown jsonb not null`
- `reasons text[] not null default '{}'`
- `evidence_refs jsonb not null default '{}'`
- `calculated_by text not null default 'system'`
- `created_at timestamptz not null`

Relationships:

- Belongs to vendor organization.
- Optionally links to a verification request.

Constraints and indexes:

- Index: `(organization_id, created_at desc)`
- Index: `verification_request_id`
- Check: `score between 0 and 100`

Soft delete: No. Snapshots are historical facts.

Buyer exposure:

- Buyers see `score`, `trust_level`, approved badges, and a simplified score summary.
- Buyers do not see `evidence_refs`, admin notes, failed internal checks, or private document metadata.

Admin exposure:

- Admins see full breakdown, reasons, evidence references, check-level status, and document metadata.

## Search And Buyer-Facing Projections

Vendor search should query only approved or buyer-safe fields:

- `organizations.name`, `slug`, `industry`, `country`, `region`, `city`
- `vendor_profiles.business_summary`, `regions_served`, `current_trust_score`, `current_trust_level`, `public_profile_enabled`

- Active `badge_assignments`
- Approved `verification_requests.status = 'approved'`

Recommended indexes for search:

- B-tree: `organizations(type, status)`
- B-tree: `vendor_profiles(onboarding_status, current_trust_score)`
- GIN: `vendor_profiles.regions_served`
- B-tree: `vendor_capabilities(service_category_id, organization_id)`
- B-tree: `badge_assignments(organization_id, expires_at)`
- Optional future full-text index on organization name, profile summary, category names, and capability names.

- `documents.rejection_notes`

- `verification_checks.admin_notes`

- `audit_events.before_data`
- `audit_events.after_data`

## Trust Scoring Model

The trust score is out of 100 and recalculated when relevant evidence changes, especially document review status, checklist status, reference validation, case study validation, and onboarding completion.

Initial weighted categories:

- Core business identity completeness: 15
- Tax and registration documents: 20
- Bank/account proof: 10
- Address proof: 10
- References validated: 15
- Case studies present or verified: 10
- Category-specific compliance documents: 15
- Responsiveness and onboarding completion: 5

Persistence rules:

- Current state is stored on `vendor_profiles.current_trust_score` and `vendor_profiles.current_trust_level`.
- Per-request score is stored on `verification_requests.current_score`, `score_breakdown`, and `score_reasons`.
- Check-level evidence is stored in `verification_checks`.
- Historical recalculation events are stored in `trust_score_snapshots`.

Badge thresholds:

- `verified`: 70+
- `trusted`: 80+
- `premium_verified`: 90+

These thresholds are catalog records in `trust_badges`, not hard-coded UI values.

## Migration Notes

- First migration should enable required extensions, especially `pgcrypto` for UUID defaults and `citext` for case-insensitive email.
- Create enums before tables that reference them.
- Create global catalog tables before tenant tables that depend on them.
- Create `organizations`, `users`, `roles`, and `memberships` early because most tables reference them.
- Add indexes in the same migration as table creation unless index build time becomes a problem.
- For large future deployments, consider concurrent index migrations separately.
- Keep JSONB limited to flexible evidence, metadata, feature flags, and audit diffs where relational modeling would add noise without query value.
